# Fellowship Priorities & Success Metrics

Source: Worldwide Venture Fellowship, Spring 2026. Led by Michael Raspuzzi.

## The floor (minimum success metrics — 100-day program)
1. **50+ user feedback sessions** (real conversations, not surveys)
2. **First paying customer** (even $1 counts)
3. **Ship production-ready prototype** with AI-native features
4. **10k+ views** on a build-in-public post (go viral 1x)
5. **Weekly channel** to engage customers directly

## Next 50 days goals (from mid-demo, April 2026)
1. Substack to 1,000 subscribers
2. YouTube build-in-public to 500 subscribers
3. First school payment by end of April
4. 10 homeschool families by end of April
5. Scale pilot from 11 to 50+ learners
6. Apply for Renaissance Philanthropy "AI for Math Fund"

## Hustling expectations
- 5 calls per week minimum
- 50-100 outreach messages per week
- Manual first 100-200 messages (learn what resonates)
- Focus on problem validation, not selling
- Demo only AFTER pain point is stated

## Critical fellowship feedback questions (agents should help answer these)
1. What are frameworks for learning? (education efficacy, data backing)
2. How do you own entire data pipeline? (student action → feedback → model improvement)
3. How can you do different things other than chatbots?
4. Do other students learn when playing other students' games?
5. Where is human in the loop?
6. What's age range?
7. How do you ensure quality of UGC competing with DuoLingo (full visual team)?
8. How does token economy work? Gaming has 90% drop-off in first 30min.

## How this affects Orchestrator priorities
- Every task should be evaluated against: "Does this get us closer to a paying customer?"
- Demo-ready features > technical perfection
- User feedback sessions > building in isolation
- One verified standard that WORKS > many broken standards
- "Fall in hate with the problem, not in love with the solution" — stay flexible on implementation

## Problem memo framework (for pitching)
Keep pitches to 5 sections, < 1 page:
1. Introduce ONE person who has the problem
2. Clear problem statement (specific, data-driven, geographically scoped)
3. Status quo and gap
4. Overview of the solution
5. Who pays, how much, and why

## Open source decision
Fellowship recommends Apache 2.0 license. Open source increases builder recruitment, velocity between fellows, and movement building. THE SECRET (learners build games) is protected by workflow design, not code secrecy.
