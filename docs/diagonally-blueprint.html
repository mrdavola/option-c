2. Glossary — Math & Game Mechanics

These terms describe how math is organized and turned into gameplay.

TermDefinition
Game MechanicThe core gameplay pattern tied to a math domain. There are 24 mechanics (e.g. "Balance & Equalize", "Split & Share", "Montessori Materials", "Singapore CPA"). Each has a stick figure animation that demonstrates the mechanic. The mechanic determines HOW the player interacts with the math.
Math RoleAuto-filled from the Game Mechanic. Describes how math fits into the game (e.g. "Add and remove to make both sides match"). The learner doesn't choose this — it comes from the mechanic.
Game EngineThe pre-built game code for a mechanic. Handles all game logic (rounds, scoring, win/lose, difficulty). There are 24 engines (one per mechanic), each built on the shared Phaser base (Boot, Victory, GameOver, HUD, game juice). The engine is the MACHINE that runs the game. Each engine file contains ONLY the gameplay code — everything else comes from the shared base. Engines are tested and guaranteed to work.
Game OptionA specific gameplay mode within a Game Engine. There are 81 game options across 24 engines — each is a genuinely different way to play the same math, not just a difficulty tweak. Example: "Balance & Equalize" has Free Balance, Mystery Side, and Chain Scales. Pedagogy engines have more: Montessori has 6 options (Golden Beads, Hundred Board, Stamp Game, etc.). The learner picks one when building their game. In code, game options use string IDs like "free-collect", "golden-beads", "inequality-grapher".
Theme ConfigA JSON object generated by AI from the learner's Circuit Board Builder picks. Contains names, colors, messages, and sprite selections that skin the Game Engine. The config is the PAINT on the machine. AI generates it in ~2 seconds.
Sprite Library30 SVG assets used in games: 10 characters (pirate, robot, astronaut, knight, chef, diver, ghost, ninja, wizard, explorer), 10 items (coin, gem, treasure chest, crystal, potion, fruit, star, shell, mushroom, key), 10 backgrounds (underwater, space, forest, castle, kitchen, cave, city, volcano, arctic, jungle). GameplayWhat the player does on screen — the interaction mechanic. Each game option has a specific gameplay: dragging weights (Balance), pouring liquid (Pour the Liquid), clicking coordinates (Battleship), etc. The gameplay is the INTERACTION SHELL. All math content comes from AI-generated Round Data specific to the standard. The gameplay interaction IS how the learner demonstrates the math. The AI math prompt appears at the top; the player uses the gameplay to answer it.
Round DataAI-generated math problems tailored to the specific Common Core standard. Each round has: a prompt (what the player sees), a target (correct answer), items (clickable numbers including answer + distractors), and a hint. Generated by Haiku at build time. Phaser EngineAll 24 game engines use the Phaser.js game framework. Each engine provides the gameplay interaction. Math content comes from AI-generated Round Data. Shared Phaser base handles: Boot, Victory, GameOver, HUD, character sprites, game juice.
Per-Standard MappingEach of the 466 moons is individually mapped to 1-4 game options that genuinely test that specific math skill. Stored in standard-game-options.ts. This prevents mismatches like a ratio standard getting a volume game. The Game Assembler reads this mapping to show only valid options.

Engine Architecture: Shared Base vs. Per-Engine Code

Every Phaser engine is split into two layers: a shared base (identical across all 24 engines) and per-engine code (unique to each mechanic and its game options). This separation means each engine file focuses purely on gameplay — all infrastructure is handled once.

Shared Phaser Base (base-phaser-template.ts)

The shared base is a function that wraps any engine's gameplay scenes into a complete, playable HTML file. Every game gets these for free:

  • BootScene — loading bar, preloads character/item/background sprites from the sprite library
  • VictoryScene — character celebration animation, particle fireworks, score display, dare text, sends game_win postMessage to parent app
  • GameOverScene — "game over" feedback with wrong-answer count, sends game_lose postMessage (triggers Hint Card)
  • HUD overlay — game title (Space Grotesk), score counter, 5 round dots, 3 life hearts, help button (?)
  • Game juice library — burstParticles(), screenShake(), comboCounter(), scorePop() available to all engines
  • Fonts — Space Grotesk (titles) + Lexend (body) loaded via Google Fonts CDN
  • Phaser config — 800×600, Scale.FIT, CENTER_BOTH, arcade physics
  • postMessage protocol — game_win and game_lose for parent app integration
  • Tutorial overlay — "How to play" screen shown before first round, with text provided by the engine

Per-Engine Code (what each engine file provides)

Each engine file (e.g. collect-manage-phaser.ts) contains ONLY the gameplay that makes that mechanic unique. It provides four things:

ComponentWhat it isExample (Collect & Manage)
GameScene class(es) The Phaser Scene(s) containing the core interaction loop — what the player sees, clicks, drags, or types each round. Each game option can be a separate scene class, or options can branch within one scene if they share enough logic. This is where the game FEELS different between options. Free Collect: items appear on screen, click to add to total, hit exact sum.
Conveyor Belt: items scroll left-to-right on a belt, tap to grab before they fall off the edge.
Split the Loot: two bins at bottom, drag items into either bin, each bin has its own target sum.
Round generation logic A function that produces the numbers, positions, and targets for each round. Takes the current round number (1–5) and difficulty curve, and returns: the target value(s), the item values to display, and their positions/timing. Must guarantee at least one valid solution exists. When AI-generated RoundData is available (math-standard-aware problems), the engine uses those instead of its own random generation. Round 1: target=12, 5 items (values 3,4,8,5,7), subset {5,7} sums to 12.
Round 5: target=48, 9 items with tighter margins and more distractors.
Intro text A short, plain-language explanation shown on the tutorial overlay before the game starts. Must tell the player: (1) what they'll see on screen, (2) what they need to do, and (3) what counts as right vs. wrong. Written for the grade level of the learner. Free Collect: "Items appear with numbers on them. Click items to collect them — your total must match the target exactly. Too many or too few and you lose a life!"
Conveyor Belt: "Items slide across a belt. Tap the ones you need before they disappear! Your total must match the target."
Help panel content A longer explanation shown when the player clicks the (?) button during gameplay. Includes: a restatement of the goal, a worked example showing a correct move, and a worked example showing an incorrect move. This is the in-game math support — not the Hint Card (which is AI-generated after a loss). "Each round shows a target number. Click items to collect them — your total must match exactly.

✅ Target is 15 → click 8 + 7 = 15 (correct!)
❌ Target is 15 → click 8 + 9 = 17 (too much!)"

Key rule: Engine files must NOT contain any HUD code, victory/game-over screens, sprite loading, font imports, postMessage calls, or Phaser config. All of that lives in the shared base. If you're writing code that isn't about how this specific game option plays, it belongs in the base.

This is important to understand: the game logic is built by us, not by AI. AI only handles the creative "skin" on top.

Intrinsic IntegrationHabgood & Ainsworth (2011) showed that math woven INTO game mechanics produces better learning than math layered ON TOP. Two tests every intrinsic game must pass (both required):
1. Discovery: A learner who doesn't already know the math can LEARN the concept by playing. The game teaches through its own mechanics.
2. Self-Revealing Truth: Correctness is shown by the physics/behavior of the game world — not a "correct/wrong" popup. Objects themselves reveal rightness.

Drag vs click is irrelevant. The test is: does the player DISCOVER the answer through the interaction, or do they ALREADY KNOW it and just deliver it?		Habgood & Ainsworth (2011)

The Diagonally Learning Flow (research-backed)

  1. Animated visual (3 seconds) — stick figure demonstrates the math concept with actual numbers. One caption sentence. Builds pictorial understanding. (CPA pictorial stage)
  2. Build the game (~30 seconds) — learner picks components in the arcade cabinet. They're invested because it's THEIR game. (Ownership + motivation)
  3. First play with Hint Card — learner plays their own game WITH the hint card visible and math explanation available. They struggle, fail, read hints, try again. Wins don't count. (Productive struggle + CPA concrete/abstract)
  4. Play for real — after practicing, learner plays without hints. Wins count toward demonstration. The math is now the gameplay itself. (Intrinsic integration)
  5. Play others' games — to master, play different games on the same concept built by other learners. Different game options = different ways to apply the same math. (Interleaving)

13. The Learning Contract — Per-Moon Specification

Decided April 14, 2026: Before any game is built or any standard is "done," it must have a completed Learning Contract. No contract, no code. The contract is Mr. Chesure's checklist for ensuring a game actually teaches its standard. Inspired by Brilliant.org's "one concept, hands-on, instant feedback" pedagogy and Rosebud.ai's "describe your game" creative model.

Core design principles (non-negotiable)

  • Split into two layers: inner = Math Mechanic (Brilliant-style, rigorous, validated), outer = Theme (Rosebud-style, AI-generated cosmetic wrapper). The math lives in the inner layer. The theme never changes how learning happens.
  • One concept per game. Don't bundle skills.
  • The LEARNER does the math, not the system. ⚠️ The system never displays a running count or running total while the learner is performing the math. Numbers appear only in (a) initial prompts, (b) multiple-choice answer options, and (c) correctness verification AFTER the learner commits. If the system is doing the counting/combining while the learner taps, the learner is just tapping, not learning.
  • Force the math, prevent the shortcut. The mechanic must make the target skill the ONLY winning action. Matching, guessing, or visual pattern tricks must be impossible.
  • Feedback is instant BUT passive. State updates show what the learner did (counters appear where tapped). They do NOT reveal math results.
  • Concrete → Pictorial → Abstract. Start with manipulable objects. Don't show symbolic representation until later grades.
  • One well-built game = many standards. A single validated mechanic covers every standard that shares its pedagogy. Target: ~20 total mechanics for 466 standards.

Visual / aesthetic principles (NEW)

Shifting from our current dark-mode arcade aesthetic to a Brilliant-inspired light/crisp/minimal aesthetic:

  • Background: off-white (#fafafa) or very pale tint — not black. Math stands out.
  • Typography: generous whitespace, clean sans-serif (Lexend for body, Space Grotesk reserved for major headings ONLY)
  • Color palette: muted accents (soft blue, gentle green, warm amber). Save bold color for THE math element (the one thing the learner should focus on).
  • Layout: lots of breathing room. Math object centered. Controls below. No UI clutter above.
  • No arcade juice: no screen shake, no particles, no flash. Gentle success states (soft checkmark, color change) not fireworks.
  • Animation: subtle, purposeful — items slide into place, feedback fades in. Nothing distracts from the math.
  • Hero / character: removed from the math mechanic entirely. Lives only in the themed build layer.
  • Chrome (score, lives, etc.): minimal and subtle — not a full top banner. Maybe just a small progress indicator.

15. Math Alignment — How We Guarantee Games Teach the Right Math

This is the backbone of Diagonally. Every design decision in this section exists to ensure that when a learner plays a game, they MUST use the exact Common Core math skill to win.

The Problem We Solved

During testing, we found a learner building a game for "Understanding Ratios" (6.RP.A.1) — but the game was a Box Packer that tested volume (L x W x H). The AI-generated hint card correctly explained ratios, but the game itself had nothing to do with ratios. A learner could "demonstrate" the standard without ever using ratio reasoning. This makes the app useless.

Our Three-Layer Guarantee

LayerWhat it doesWhere in code
1. Per-Standard MappingEvery single one of the 466 moons is individually mapped to 1-4 game options that genuinely require that specific math skill. A math teacher audited all 466 mappings. No keyword guessing — hardcoded lookup.standard-game-options.ts
2. AI Rounds OnlyEngines are pure interaction shells. ALL math content comes from AI-generated rounds tailored to the specific standard. No engine generates its own random math. The shared getRound() function ensures every scene reads from AI_ROUNDS.base-phaser-template.ts (getRound), all 81 game scenes
3. Validation + RetryAfter AI generates rounds, the system checks if prompts match the standard description. If match is too low (<10%), retries once with a stricter prompt. If retry fails, flags for admin review.generate-engine/route.ts

Rationale: Why Per-Standard Mapping Instead of Domain Matching

We tried three approaches in sequence:

  1. Keyword matching (first attempt): matched standards to mechanics based on keywords in the description. Failed because keywords overlap across domains — "ratio" appears in geometry AND proportional reasoning, "area" appears in measurement AND operations.
  2. Domain-level mapping (second attempt): mapped each domain code (like "MD" for Measurement & Data) to 2-3 mechanics. Failed because one domain contains very different concepts — "MD" includes volume, time, money, and line plots, each needing different game mechanics.
  3. Per-standard mapping (final): every single standard mapped individually by a math teacher. This is the only approach that works because each standard describes a UNIQUE skill.

19. Strategy — Open Source, Data Moat & Competitive Advantage

The Three-Part Strategy (decided April 16, 2026)

Based on fellowship advice from Michael Raspuzzi, Dylan, and a guest advisor.

Part 1: THE SECRET (competitive advantage)

NEVER reveal publicly that learners build the games. This is Diagonally's core innovation. Publicly, we are "a math games platform." The fact that learners are both creators and players is the moat.

Part 2: Open Source Strategy — Modular

Key insight (from Dylan): Open source PARTS of the app, not all of it. Keep the proprietary workflow private.

Open Source (Apache 2.0)Keep Private
  • Game engines (Number Frames, etc.)
  • Agent definitions & cross-check system
  • HTML sanitizer + security scanning
  • Galaxy 3D map UI

Why open source the engine: Recruit builders, increase velocity between fellows, build a movement in education. Fellowship recommends Apache 2.0 license.

Why keep workflow + data private: The build step, peer play, and learning data are what make us unique. Cloning the code doesn't give you the workflow insight or the dataset.

Part 3: The Proprietary Dataset — Our Real Richness

Key insight (from fellowship guest advisor): "Your richness is the proprietary dataset of how learners learn."

Every learner interaction generates data nobody else has:

  • Game performance: Which rounds learners get right/wrong, time per round, hint usage, retry patterns
  • Misconception mapping: Which wrong answers are most common per standard (reveals what kids actually misunderstand)
  • Builder decisions: What real-world scenarios learners write, which game designs pass/fail criteria
  • Peer learning: Do learners who play peer-created games learn differently than professional games?
  • Engagement patterns: Where learners drop off, what brings them back, session lengths by age
  • Shortcut attempts: How learners try to game the system (reveals design weaknesses)

The flywheel: More learners → more data → smarter games → better outcomes → more schools adopt → more learners. This is the DuoLingo model applied to math.

This data is never open sourced. The Security Guard monitors the boundary. The Tracker ensures we capture everything.

20. Project History & Lessons Learned

Full narrative of what we built, what went wrong, and what we learned. Written April 16, 2026. This is the institutional memory — any agent or team member can read this and understand the full context.

Full document: docs/project-history-and-lessons.md

The 7 Failure Modes (Summary)

  1. Games didn't teach the right math — keyword matching mapped standards to wrong game options (ratio → volume game). Fixed with hardcoded per-standard mapping.
  2. Fake intrinsic problem — games LOOKED interactive but the player action didn't map to the math operation. Mystery-side taught counting, not algebra. Dragging ≠ intrinsic.
  3. AI rounds didn't match standards — AI generated generic math or used theme words literally ("skeleton" → spooky content). Fixed with hardcoded rounds.
  4. Theme layer fought the math — "Desert Athlete" theme on a scale/blocks game made no sense. Fixed by separating math mechanic from theme (theme muted for now).
  5. System did the math, not the learner — running counts, animated counting, live totals meant the learner just watched. Fixed: no counts displayed during learner's counting.
  6. Number pad allowed brute force — wrong answers faded out, leaving fewer options. Fixed: wrong answers shake, pad hides, learner recounts.
  7. Too many games, too little quality — overnight rebuild of 87 games produced code that compiled but didn't teach. Fixed: one moon at a time, Learning Contract workflow.

10 Key Insights

  1. Mr. Chesure's 2 Tests: Discovery (learn by playing) + Self-Revealing Truth (physics shows correctness).
  2. The 3 Criteria: Real-world math + math IS gameplay + must know math to win.
  3. Fake intrinsic is more dangerous than quiz-wrapper — creates illusion of learning.
  4. The learner does the math. System never counts, sums, or computes on behalf of learner.
  5. Concrete before symbolic. Show objects before equations (K–2 especially).
  6. One well-built game = many standards. ~22 games for 466 standards via parameter changes.
  7. Don't invent pedagogy — adapt it. Progressions Docs for what. Math Learning Center for how.
  8. Test one moon perfectly before scaling. Quality over coverage.
  9. Cross-checking prevents errors. Every agent's work verified by another.
  10. Learner as builder is the unique value — paste HTML games, peer critique, guide approval.

What We Tried That Didn't Work

  • Overnight rebuild of 87 games — parallel agents wrote code but couldn't judge pedagogy. Bulk ≠ quality.
  • AI-invented game mechanics — creative but pedagogically unsound. AI should ADAPT proven methods, not invent.
  • Cross-referencing multiple sources — too many sources = confusion. Use ONE primary source per need.
  • Showing equations as K prompts — digits let learners match without understanding. Dot clusters instead.
